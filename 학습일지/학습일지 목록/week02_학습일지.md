## 📚 이번 주 학습한 것

### 1. Structured output — 자연어와 애플리케이션 사이의 계약

사용자의 자연어를 그대로 다음 로직에 넘기면 일정 종류나 날짜를 안정적으로 읽기 어렵다. 이번 주에는 `StructuredRequest`와 `StructuredRequestBatch`를 정의해 요청을 `personal_schedule`, `group_schedule`, `todo`, `reminder`, `unknown` 중 하나로 분류하고, 제목·날짜·시간·참석자 등을 검증 가능한 객체로 변환했다.

Pydantic의 `Field(description=...)`은 단순한 문서가 아니라 모델이 각 필드의 의미와 형식을 판단하는 지시가 된다는 점이 인상 깊었다. 날짜와 시간이 불명확할 때 값을 지어내지 않고 `None`이나 빈 목록을 사용하도록 명시한 것도 안전한 구조화에 중요했다.

### 2. 분류 기준과 우선순위 — 재현성은 프롬프트의 구체성에서 나온다

처음에는 가능한 `kind` 값만 알려 주고 판단을 모델에 맡겼다. 같은 입력에서도 결과가 달라지고 개인 일정과 그룹 일정의 경계가 모호해질 수 있다는 리뷰를 받고, 다음 판정 순서를 프롬프트에 추가했다.

1. 특정 시점에 알려 달라는 요청은 `reminder`
2. 완료할 작업이며 일정 생성이 아니면 `todo`
3. 사용자 외 참석자가 있으면 `group_schedule`
4. 다른 참석자가 없으면 `personal_schedule`
5. 확정할 수 없으면 `unknown`

또한 `members`에는 사용자 자신과 단순히 언급된 사람을 제외하고 실제 참석자만 담도록 했다. 선택지뿐 아니라 경계 조건과 충돌 시 우선순위까지 정의해야 결과가 반복 가능해진다는 것을 배웠다.

### 3. 출력 형식과 역할 범위 제한

모델이 올바른 JSON 뒤에 설명이나 두 번째 객체를 덧붙이면 파서가 실패할 수 있었다. 최종 응답은 `StructuredRequestBatch` 하나만 반환하도록 하고, 인사말·Markdown 코드 블록·추가 JSON을 금지했다. Week 2에서는 구조화까지만 담당하고 SQLite 저장, RAG 검색, 외부 일정 조율은 하지 않도록 책임도 분명히 나눴다.

## 🧱 막혔던 지점 & 해결 과정

- 문제: 동일한 입력의 `kind`가 달라지거나 개인/그룹 일정이 일관되지 않게 분류될 수 있었다.
- 시도: 필드 설명을 자세히 작성했지만 값별 판정 기준만으로는 충분하지 않았다.
- 해결: 분류 기준, 적용 순서, `members` 판정법을 시스템 프롬프트에 명시하고 각 요청 유형을 반복 실행했다.
- 검증: 문법 검사와 smoke test, `git diff --check`를 수행하고 개인 일정·그룹 일정·할 일·알림·애매한 질문을 반복 입력해 파싱과 분류 결과를 확인했다.
